# Vehicle Node

This subsystem builds the Raspberry Pi vehicle-node image described in `docs/vehicle-node-prd.md`.

What is implemented here now:

- strict per-vehicle TOML profile validation with unknown-field rejection
- a Docker-based image build entrypoint for Raspberry Pi OS Lite image personalization
- rendered runtime settings on the boot partition
- first-boot provisioning through Raspberry Pi OS `custom.toml`
- NetworkManager-based Wi-Fi provisioning with static-IP keyfile install
- a pinned `mediamtx` RTSP camera service using the built-in `rpiCamera` source
- a dry-run Python vehicle daemon that accepts the new 8-channel RC packet, enforces session and arming rules, and emits telemetry
- Python unit tests for profile validation, runtime settings, packet parsing, session state, and telemetry behavior

What is still hardware-gated:

- integrated appliance boot validation on a flashed Pi (`#13`)
- real ESC output timing and drivetrain characterization (`#14`)

## Known-Good Dry-Run Control Setup

The current known-good dry-run control path is:

- `fruit-vehicle.service` active on the Pi at boot
- UDP control listener on port `9000`
- telemetry return port `9002`
- `vehicle_id = 100` for the current `rcmower` profile
- active-session lock, arming, and degraded transition verified on hardware

Useful checks on the Pi:

```bash
systemctl status fruit-vehicle.service
journalctl -u fruit-vehicle.service -n 100 -o cat
ss -lunp | grep 9000
```

Useful packet-sender check from another machine:

```bash
python3 vehicle_node/tools/send_test_rc_packet.py --host 192.168.86.18 --vehicle-id 100 --arm 1.0
```

## Known-Good RTSP Setup

The current known-good vehicle-node camera path is:

- Raspberry Pi 3 A+
- Raspberry Pi OS Lite 32-bit Bookworm image generated by this subsystem
- CSI camera attached locally to the Pi
- `mediamtx v1.18.2`
- RTSP URL `rtsp://192.168.86.18:8554/stream`

Useful checks on the Pi:

```bash
systemctl status fruit-mediamtx.service
journalctl -u fruit-mediamtx.service -n 100 -o cat
libcamera-hello --list-cameras
```

Useful checks from another machine:

```bash
ffplay rtsp://192.168.86.18:8554/stream
```

## Layout

- `profiles/examples/` checked-in schema examples
- `profiles/local/` ignored real per-vehicle profiles
- `image/` Docker builder, locked upstream image metadata, and image patch scripts
- `src/fruit_vehicle/` shared Python code for profile parsing, runtime settings, and the dry-run daemon
- `tests/` Python tests for the vehicle-node subsystem

## Build

The high-level build command is:

```bash
./vehicle_node/image/build.sh --profile vehicle_node/profiles/local/my-vehicle.toml
```

By default the build requires a clean git tree so the image artifact can be tied to a known source revision. Use `--allow-dirty` to override that explicitly.

Outputs land in `vehicle_node/dist/`:

- `<profile>-<source-id>.img`
- `<profile>-<source-id>.img.xz`

The builder downloads the exact Raspberry Pi OS Lite input pinned in `image/artifacts.lock.toml`, stages runtime files and config into the image, validates the result by inspecting filesystem contents, and compresses the finished image.

## Tests

Run the Python vehicle-node tests with:

```bash
python3 -m unittest discover -s vehicle_node/tests -p 'test_*.py'
```

## Profile workflow

Start from `profiles/examples/tracked.example.toml`, then create a real local profile under `profiles/local/`.

Public keys referenced by a local profile should live under `profiles/local/keys/` and be referenced with relative paths so the Docker builder can see them inside the repo mount.
